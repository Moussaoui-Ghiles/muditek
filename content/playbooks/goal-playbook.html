The core shift: prompting vs assigning

A regular prompt asks an agent for the next response. You read what comes back, decide if it's right, and push the agent to the next step. You are steering every turn.

/goal flips that. You define the finished result, submit it once, and the agent drives toward it.

Prompting — you drive
you write a prompt
agent answers
you read and judge
you push the next step
repeat, every turn
You are the loop. Nothing moves without you.
Assigning — the agent drives
you define "done"
you submit /goal once
agent plans the steps
agent works toward the target
you walk away
You set the target. The agent runs the loop.
"The shift is from prompting (you driving) to assigning (the agent driving toward a target you defined)."

In practice this turns the agent from a reactive chatbot into a background worker that can break a task into smaller steps, use tools, write and run code, iterate, self-correct, and keep going with minimal supervision until the work is done.

One thing to get right early

The real primitive lives inside an interactive worker session. You launch the tool, you submit /goal, and you walk away. Putting the word "goal" inside a normal one-shot command (for example codex exec 'goal: build the app') is still just a prompt with a label, not an assigned goal.

The lifecycle of a goal

Once you assign a goal, it stays active until one of these happens:

  • Achieved — the success criteria are met.
  • Paused — you pause it manually.
  • Blocked — it hits something it can't resolve.
  • Cleared — you clear it to start fresh.
  • Out of budget — it runs out of allowed turns.

This is why a goal can run for minutes, for hours, or overnight. It is not waiting on your next message. It is working against the definition of done you gave it.

Steer without micromanaging

Use /goal status to check in and /subgoal to adjust direction mid-run. You do not need to stop the agent to redirect it.

How to write a goal that finishes

The quality of the result depends heavily on how well you define the goal. This is the highest-leverage skill in the whole system.

A strong goal is:

  • Specific and measurable — there is no ambiguity about what to build.
  • Backed by clear success criteria — the agent can tell when it is done.
  • Well-scoped — not so broad that "done" is impossible to define.
"The more clearly you can describe the final outcome and how to verify it, the better the agent will perform."

The "done means" pattern

The strongest goals state the finish line as checkable conditions:

goal · build from spec
/goal Build the app described in SPEC.md. Done means tests pass,
build passes, README is accurate, and git status only shows
relevant project files.

Notice every condition is verifiable: tests pass, build passes, README accurate, clean git status. That is what makes the goal completable instead of open-ended.

Strong vs weak goals

Weak — avoidWhy it fails
"Make something cool"No outcome, no success criteria.
"Improve my code"Unmeasurable. "Improved" by what standard?
"Work on the project"No scope, no finish line.
Strong — useWhy it works
"Create a fully functional Flappy Bird clone in HTML5 with physics, keyboard and mouse controls, scoring system, and collision detection. The game must run on localhost and all core mechanics must work."Specific features plus a verifiable run state.
"Build a clean multi-page website with a homepage, features, and pricing. Make it responsive and pass basic Lighthouse checks."Named pages plus a measurable quality bar.
"Refactor the main processing module, improve performance by 30%, add proper error handling, and ensure all tests pass."A numeric target plus checkable conditions.

The recommended workflow

A repeatable five-step loop for any goal:

  1. Provide context first. Give the agent information about your project, tech stack, folder structure, and previous decisions before you assign anything. Context is what makes the output match your actual project.
  2. (Optional) Generate strong goals. Paste this meta-prompt to have the agent propose goals worth running:
prompt · generate strong goals
Based on what you know about me and my projects, suggest 3 strong /goal
ideas that would run for a long time and create the most value.
  1. Launch the goal. Write /goal followed by a detailed description.
  2. Manage the process. Use /goal status to check progress, /subgoal to adjust direction, and pause or resume as needed.
  3. Review the result. The agent returns completed work, a summary, or an explanation if the goal could not be fully achieved.

The three roles in an agent team

/goal becomes far more powerful when more than one tool speaks it. The tools can change, but the roles stay the same.

Orchestrator
e.g. Hermes
Owns the control loop. Decomposes the task, picks the right worker for each job, runs background processes, tracks dependencies, verifies the final result, and writes the summary you actually read.
Builder
Codex
Takes a spec and produces working code. Implementation is the bottleneck this role solves.
Reviewer
Claude Code
Reads what the builder produced and finds what is wrong: spec compliance, safety, error states, security holes. Correctness is the bottleneck this role solves.

Which tool for which role

ToolRoleStrength
Codex (OpenAI's coding CLI)BuilderImplementation, especially from a clear spec. /goal is how you hand it that spec.
Claude Code (Anthropic's coding CLI)ReviewerThe inverse of building: finding what is wrong with code that looks right — spec compliance, safety issues, error states, security holes. /goal is how you point it at code and ask for a review.
Orchestrator (e.g. Hermes)OrchestratorCoordinates work between the coding workers above. /goal is how it hands off tasks to the right tool, and how you tell it what you want.

Why this matters

Three different teams converged on the same /goal instruction format even though the tools share nothing else. That convergence is what makes it possible to compose them. If each tool had invented its own job-handoff format, no orchestrator could route between them.

The whole point

One message turns into a pipeline across multiple tools, and you watch the whole thing move across one board. You stop sitting in a terminal waiting for one agent to finish and start managing a queue of work with visible state.

"The board is what /goal becomes when there's an orchestrator on top of it. Every goal has a card, every card has a status, every handoff leaves a trail."

The verification rule

This is the single most important rule in the entire system.

The rule that makes it a contract
"Don't trust the worker's self-report as final. Trust the verifier."

An orchestrator should never accept the worker's word that the job is done. After a builder marks a build complete, the orchestrator runs the checks itself:

verify · run it yourself
npm test      # actually run the tests
npm run build  # actually run the build

Why this is non-negotiable:

"Coding agents are confident. They'll tell you the build passes when the build was never run. They'll tell you tests pass when they wrote tests that never executed. The verifier closes that gap."

The takeaway:

"Without verification, /goal is just a fancier prompt. With verification, it becomes a contract."

Action: always bake the verification commands into your SPEC.md and done-criteria, and make sure something other than the builder actually runs them.

Setup: treat installation as a goal

Once you have an orchestrator running, you should not set up new tools by hand.

The first time you need an additional coding worker on the machine that runs your orchestrator, do not install it manually. Send the orchestrator a message asking it to install the tool and log you in. It handles the rest.

"You don't type install commands. Setup is just another goal."

If you do not have an orchestrator yet, follow the official install pages for the individual coding tools to get started. But the point of having an orchestrator is that mechanical work stops being yours.

Combine tools deliberately

Running an orchestrator together with a dedicated builder and a dedicated reviewer produces better results than any single tool working alone.

Running multiple goals in parallel

You can run multiple goals at once, but you cannot point multiple coding workers at the same files without thinking about it first.

The default: one main builder per repo

Add parallelism only across clear boundaries where two workers cannot step on each other:

  • Different repos
  • Different branches
  • Git worktrees
  • Separate packages
  • Docs vs. code
  • Tests vs. implementation

The bad pattern

Three workers all editing the same file in the same repo. You get conflicts, partial overwrites, and one worker silently undoing another's work.

The better patterns

  • One writer at a time on any given file. The builder writes, the reviewer only reads, and fix goals stay scoped to the fix.
  • Or run several builders in several worktrees on competing approaches to the same problem, and let the orchestrator pick the best result.
"Without the board, parallel background workers become terminal chaos."

Best practices and common mistakes

Best practices

  • Always make goals measurable with clear success criteria.
  • Use /subgoal actively to steer the agent while it runs.
  • Increase the turn budget for long-running tasks (for example: hermes config set goals.max_turns 500).
  • Combine your orchestrator with dedicated builder and reviewer tools.
  • Run complex goals overnight.

Common mistakes

  • Using vague goals with no success criteria.
  • Intervening too often instead of letting the agent work.
  • Starting without enough project context.
  • Making goals too broad or open-ended.

When to use /goal, and when not to

Use it for

  • Complex, multi-step tasks (building apps, refactoring, research).
  • Work that benefits from iteration and self-correction.
  • Tasks you can leave running for hours.
  • Situations where you want to delegate rather than micromanage.

Skip it for

  • Simple or quick tasks.
  • Situations where you need full control over every single step.
  • Work where you have not yet clearly defined the desired outcome.

Quick-start checklist

  • Give the agent your project context (stack, structure, decisions).
  • Write a goal with a clear, verifiable "done means …" definition.
  • Put the verification commands inside the goal or SPEC.md.
  • Assign it with /goal, then walk away.
  • Check in with /goal status; steer with /subgoal. Do not micromanage.
  • Make sure something other than the builder actually runs the tests and build.
  • For parallel work, isolate workers by repo, branch, or worktree — never two writers on one file.
  • Review the final summary against your original success criteria.
