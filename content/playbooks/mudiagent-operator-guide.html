01 What MudiAgent is

An autonomous agent that gets more capable the longer it runs

Most AI tools answer questions. MudiAgent runs your workflows end to end. It navigates your browser, executes terminal commands, schedules cron jobs, monitors your inboxes, drafts the work, and posts the result to wherever you work: Telegram, Discord, Slack, or the email thread you are already in.

The longer version: MudiAgent is a framework that turns a model into a persistent operator. It has its own memory that survives between sessions. It writes its own skills as it works. It ships with 123 skills already built in. It lives wherever you put it, on your laptop, in a Docker container, on a VPS, or in a serverless runtime. And you can talk to it through more than 20 surfaces.

The shape of it

If you have used Claude Code, MudiAgent is the same shape with a different philosophy. It is the batteries-included option: opinionated defaults, productive on day one with minimal setup, the agent does more of the thinking for you. Lower-level frameworks give you primitives and explicit control instead, where the agent does exactly what you told it and nothing more. Both are valid. The bundled defaults compound. Every project you start is one where the agent already knows how to do a hundred things before you write a line of configuration.

What you get out of the box

  • 123 bundled skills before the agent writes one of its own
  • 70+ built-in tools in the gateway, plus 300+ models through one subscription
  • 6 deployment targets: local, docker, ssh, daytona, singularity, modal
  • 20+ messaging surfaces: telegram, discord, slack, email, voice
02 How it works

Every agent has a brain, a personality, and a skillset

A brain

Memory lives at ~/.mudiagent/memories/. Two files, MEMORY.md and USER.md, inject at session start. Your voice rubric, your brand notes, your customer language, last week's corrections, all of it loads before the first prompt. Sessions are stored in SQLite, and recall across sessions is full-text searchable.

A personality

The SOUL.md file is where the vibe lives. Concise. Sarcastic. Blunt. Formal. Fast or thoughtful. You can spin up six agents and give each one a different soul on the same brain underneath. One is your outbound rep with a closer's energy. Another is your researcher who likes long sentences. Another is your assistant who keeps everything short.

A skillset

123 skills out of the box: GitHub PRs, Obsidian, Google Workspace, Linear, Notion, Perplexity, deep research, browser control, web scraping, vision, voice, scheduling. Then the closed learning loop: as the agent works, it writes new skills along the way. Your own skills library grows on top of the 123 without you having to write any of them.

Built-in skills connect to

What the agent talks to

  • The tool gateway: one subscription, 300+ models, plus web scraping and browser automation built in
  • MCP integration: any external service that speaks Model Context Protocol becomes a tool the agent can use
  • 20+ messaging surfaces: telegram, discord, slack, email, voice, plus the CLI itself
Talk to it on

Where the agent lives

  • Your laptop (local)
  • A Docker container (isolated, portable)
  • An SSH session on a VPS, so it runs even when your laptop is closed
  • Daytona, Singularity, or Modal, if you do not want to manage infrastructure
The rule for day one

Do not try to write your own skills on day one. Run real work, let the agent watch, and let the harness write the skills. You build a custom skill library faster by working than by writing prompts.

03 What runs on MudiAgent

Marketing infrastructure, with the occasional internal tool

Common workloads teams run today:

  • A personal assistant that handles business and private, lives in Telegram, flags the few emails worth reading every morning, schedules reminders, and summarizes missed meetings
  • A prototyping bench where new flows (lead magnet, ad creative review, content sprint) run against real work for two or three runs before being promoted
  • Specialized agents for SEO, outbound, design review, and content writing, each with its own soul and its own scope
  • A company brain that monitors Slack, chats, emails, transcripts, and voice memos and makes all of it queryable, so a question like what was said to a client about pricing last month gets answered in seconds
  • An SEO agent that runs the full pipeline from keyword seed to published article in one container, 21 steps, no human in the middle until the final review
  • A content distribution agent that takes a piece of long form and atomizes it across LinkedIn, X, and Threads with platform-specific hooks
  • An orchestrator agent that produces no work itself and just routes requests to the right specialist

Zooming in on the SEO agent

The SEO agent maps cleanest to the architecture in the rest of this guide. Five layers, all inside one Docker container, 21 steps from keyword seed to published article. In the terminal the pipeline looks like this:

[research + ideate]
01 keyword seed
02 serp snapshot
03 competitor extraction
04 intent + format analysis
05 content + visual gap
06 internal validation
07 external validation

[production]
08 angle + positioning brief
09 visual strategy brief
10 outline
11 draft
12 image gen
13 flowchart gen
14 visual qa
15 article qa

[distribution]
16 publish prep
17 schema
18 internal linking
19 syndication
20 analytics setup
21 monitoring

The layers above this pipeline:

  • Company brain at the top: vision, brand, audience, products. Every agent reads from this
  • Orchestrator: takes the topic or keyword seed and routes it to the SEO agent
  • SEO brain: ranking playbook, voice rules, content formats, visual style guide, success criteria per format. All SEO-specific context lives here
  • Three sub-agents inside the SEO agent, each handling one phase: research and ideate, production, distribution
Why one container, not three

SEO work is sequential. Research feeds the brief, the brief feeds production, production feeds distribution. Every step needs memory of what was decided upstream. Splitting into three containers means shuttling state across boundaries, which gets expensive and breaks the chain. One container holds all three sub-agents. They share env, memory, and tools. Sub-profiles switch context per phase.

Every other specialized agent runs on the same template. Clone the SEO agent template, swap the brain (SEO brain to outbound brain, or design brain, or support brain), and you have a new agent for any function with the same five-layer shape. The layers are not decoration. They are the reason the agent does not lose context as the work gets specialized. The company brain stays stable while the worker iterates. The brain layers make the worker disposable.

04 From one agent to a full fleet

The mental model before the levels

The setup has four parts:

  • You are the operator. You have direct access to every part of the system.
  • The agent control room is the side control plane. It is not an agent you chat through. It is a folder at /root/vps-agents that documents and governs the whole fleet.
  • The agents are the workers. Some are specialists (seo, dev, cmo, ops). One of them can optionally be an orchestrator.
  • The agent task bus is an optional handoff desk that sits between the orchestrator and the specialists. You only need it once you have an orchestrator in play.

The storage split matters more than people think

/root/vps-agents        → control room: docs, rules, runbooks,
                          architecture. no raw secrets, ever

/srv/<agent-name>/data   → live runtime: secrets, memory, skills,
                          sessions, crons. where each agent lives

The control room is the brain that defines the system. The live runtime is the body that runs it. You can rebuild the body from the brain. You cannot rebuild the brain from the body.

Level 1

One agent

You have one agent. That is it. The control room can still exist (recommended) but it only documents that one agent. Fill in SOUL.md with the voice you want, MEMORY.md with the stable facts about your business, and USER.md with the stable facts about you. Connect it to Telegram or Discord so it lives where you do. Start using it on real tasks. Let it touch your tools. Let it write its own skills along the way.

Best for: initial setup, your personal agent, root install documentation, simple Docker migration.

Level 2

Direct specialist agents

You have multiple specialized agents, but you still talk to each one directly. No orchestrator yet.

you  →  mudiagent-life
you  →  mudiagent-seo
you  →  mudiagent-dev
you  →  mudiagent-cmo

the control room documents all of them.

The trap to avoid here is reaching for an orchestrator before you have proven your specialists are useful. Spin up two or three, run them directly, and only add an orchestrator when you find yourself wanting one front door.

When to spin up a new agent

needs its own credentials             → new agent
needs its own long-term memory        → new agent
ongoing repeated work, separate role  → new agent
otherwise                             → stay with what you have

The bad pattern is one mega-agent with every credential and every memory layer mashed together. You lose isolation, you lose the ability to revoke access cleanly, and the agent gets confused about which voice to use.

Best for: clear role separation, testing which agents are useful, avoiding premature orchestration, keeping credentials scoped per agent.

Level 3

Orchestrator and specialists

You add an orchestrator as a front door. You can still talk directly to specialists, but the orchestrator can route work and synthesize results. It reads the control room to know which agents exist, what each does, where task queues live, what requires approval, and which actions are forbidden. It does not need to ask you any of that, it reads it.

The orchestrator is the moment your setup stops being a collection of agents and starts being a team. It is also the moment the control room earns its keep, because the orchestrator is only as good as the docs it reads.

Best for: cross-functional work, delegation, summary and synthesis, one main interface for multi-agent workflows.

Level 4

Automated agent team

Same shape as level 3, but with recurring workflows and stronger automation. Weekly SEO reports run on cron. Server health checks fire daily. Backup verification runs without you asking. Cross-agent business workflows kick off on schedule. This is what a marketing department in your terminal looks like. It does not need you to start the day. It shows up to work on its own, files reports, checks itself, and only pings you for the decisions that need taste.

Best for: weekly SEO reports, content operations, server health checks, backup verification, cross-agent business workflows.

The control hierarchy

One principle to keep in your head as you climb the levels. The control room is for config, docs, runbooks, and governance. It is the admin panel for the fleet, including the orchestrator. It is not where you go to do work. For work, you talk to the agents directly: a specialist when you know which agent owns the job, or the orchestrator when you want one front door to route across specialists.

05 Setup

Point a coding agent at the template

Now you understand the architecture. The control-room template holds the exact structure described above, plus the skills your agent needs to set it up for you. You can clone it manually, but the point is that you do not have to. If you have Claude Code or Codex on your laptop, the agents do most of the work after you hand over a Hetzner API key.

The automated flow

you  →  generate a Hetzner API key
         (sign up, generate a token, drop it in your .env)
         ▼
agent → create-vps skill
         spins up a box, generates an SSH key,
         writes the alias so `ssh mudi` works
         ▼
agent → setup-control-room skill
         installs Node, Docker, Claude Code, Codex CLI,
         MudiAgent, then clones the template to the VPS
         ▼
you  →  finish interactive auth on the VPS
         (claude /login, codex, mudiagent)
         ▼
agent → agent-control-room skill
         registers your first agent in the docs,
         fills in the runbook, sets up the env-map
         ▼
you are at level 1 with a documented agent

Within ten to fifteen minutes you have a fresh VPS with the right tooling installed, the control room cloned on the VPS, the bundled skills linked in, one agent registered with a runbook and env-map, and an SSH alias on your laptop so ssh mudi connects instantly.

06 Prototype to production

You cannot write a production agent from scratch. You grow one.

Most workflows do not start as production ones. They start messy. A flow that runs SEO research, drafts an article, schedules it, and posts it does not exist in your head fully formed. You discover it by running it. MudiAgent is the prototyping environment for this. The four-step path:

  • Prototype in MudiAgent. Open your main agent, describe what you want to happen, and let it try. It will get most of it wrong on the first run. That is fine.
  • Run it two or three times against real work, correcting drift each time. The harness watches every correction and starts writing the skill as it learns the shape. By run three the agent is doing most of what you want without coaching.
  • Fine-tune in a dedicated workspace. Pull the workflow into a separate workspace, tighten the prompts, lock the routing, add error handling, decide what runs on cron and what is triggered.
  • Deploy to a VPS on a schedule. Once it survives a week of real runs without babysitting, push it to its own container, set the cron, walk away.
prototype in mudiagent
        → fine-tune in a dedicated workspace
                → deploy autonomous on a VPS
07 The models you run

The framework is fixed. The model is your choice.

Through the tool gateway you can route to 300+ models from one subscription, switching per agent or per task. A practical split:

  • Claude Opus 4.7 for the creative work: copywriting, voice, hook generation, content drafting, anything where taste and writing quality matter
  • Codex (GPT 5.5) for the structured work: coding, planning, multi-step workflows, browser automation, scraping, anything where the steps need to be tight and the output predictable

Run both. One writes, the other builds and plans. If you can only run one, the answer depends on the work: heavy on content and copy starts with Claude Opus 4.7, heavy on infrastructure and engineering starts with Codex. You can always add the second later through the same gateway.

08 Honest trade-offs

Where this still breaks

  • The bundled defaults are also opinions. MudiAgent ships with strong defaults for how memory works, how skills get written, and how the agent uses tools. That is the whole pitch. But it also means if you want primitives with explicit control over every step, it will feel heavy. Pick the tool that matches your philosophy.
  • Levels 3 and 4 have a real learning curve. Docker, VPS, SSH, the control room folder structure, the orchestrator skills. None of this is install and go. Do not jump to level 3 if you are not already running at level 1 daily.
  • The model still matters. The framework makes a good model great. It does not make a small model into a strategist. Use the strongest models you can afford for the work that matters (your orchestrator, your strategy agent, your brain) and drop to cheaper models for the work that does not (research scraping, draft generation, batch processing).

None of this is magic. It is a framework that pays back because the memory persists, the skills accumulate, and the agents stay scoped. Apply it to the wrong-sized model and you get a confused team. Apply it to the right one and you get a team.
