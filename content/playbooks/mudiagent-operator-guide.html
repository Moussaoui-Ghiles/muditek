What you get from this guide

Most AI tools answer questions. MudiAgent runs your workflows end to end. It navigates your browser, executes terminal commands, schedules cron jobs, monitors your inboxes, drafts the work, and posts the result to wherever you live. Telegram. Discord. Slack. The email thread you are in right now.

This guide is the architecture: brain, soul, skillset, in one folder. The use cases that actually run on this stack. The four-part mental model (you, control room, agents, optional task bus) and the four levels of setup. The prototype to production methodology. The honest trade-offs and where it still breaks.

01 — FOUNDATIONS

What MudiAgent is

Short version: an autonomous agent that gets more capable the longer it runs.

Longer version: MudiAgent is a framework that turns a model into a persistent operator. It has its own memory that survives between sessions. It writes its own skills as it works. It ships with 123 skills already built in (GitHub workflows, Obsidian, Google Workspace, Linear, Notion, Typefully, Perplexity, deep research, plus 100+ more). It lives wherever you put it: your laptop, a Docker container, a VPS, a serverless runtime. And you can talk to it through 20+ surfaces.

Take

It routes tasks to the right model based on complexity and cost. Learns your voice and preferences over time. Handles context switching without losing the thread.

MudiAgent vs. a coding assistant vs. an agent primitive

If you have used Claude Code or other terminal coding agents, MudiAgent is the same shape with a different philosophy.

Coding assistant
A pair programmer

Lives in your terminal next to your code. Best for sitting down, opening an editor, and drafting code with you all day. Day-driver for desk work.

MudiAgent
Rails. Productive day one.

Opinionated defaults, batteries included, the agent does more thinking for you. Lighter, faster, focused on the self-improvement loop. Built for work that runs without you watching.

Agent primitives
Linux. Explicit control.

Primitives, guarantees, the agent does exactly what you told it to and nothing more. The right choice if you want explicit control over every step.

Both directions are valid. We run MudiAgent because the bundled defaults compound. Every project starts with the agent already knowing how to do 100+ things before a line of configuration. That head start is worth it.

Why it matters: the bundled defaults are the whole pitch. They are also opinions. Pick the tool that matches your philosophy.
02 — ARCHITECTURE

A brain. A soul. A skillset. One folder.

Every MudiAgent has three things.

A brain

Memory lives at ~/.mudiagent/memories/. Two files, MEMORY.md and USER.md, inject at session start. Your voice rubric, your brand notes, your customer language, last week's corrections, all of it loads before the first prompt. Sessions are stored in SQLite. Recall across sessions is full-text searchable.

A soul

SOUL.md is where the vibe lives. Concise. Sarcastic. Blunt. Formal. Fast or thoughtful. You can spin up six agents and give each one a different soul, same brain underneath. One is your outbound rep with a closer's energy. One is your researcher who likes long sentences. One is your assistant who keeps it short.

A skillset

123 skills out of the box: GitHub PRs, Obsidian, Google Workspace, Linear, Notion, Typefully, Perplexity, deep research, browser control, web scraping, vision, voice, scheduling. And the closed learning loop: as the agent works, it writes new skills along the way. Your library grows on top of the 123 without you having to write any of them.

What the agent can talk to

  • Tool gateway: one subscription, 300+ models, plus web scraping and browser automation built in.
  • MCP integration: any external service that speaks Model Context Protocol becomes a tool the agent can use.
  • 20+ messaging surfaces: Telegram, Discord, Slack, email, voice, plus the CLI itself.

Where the agent can live

  • Local: your laptop. Simplest start.
  • Docker: isolated, portable, the way we run ours.
  • SSH on a VPS: so it runs even when your laptop is closed.
  • Serverless: if you don't want to manage infrastructure.

The closed learning loop

The closed learning loop is what makes this different from a smart chatbot. The agent watches itself work, writes new skills as it learns the shape of your work, refines its memory periodically, and recalls past context across sessions using a mix of full-text search and LLM summarization. You do not have to re-teach it next week.

Day one rule

Do not try to write your own skills on day one. Run real work, let the agent watch, and let the harness write the skills. You build a custom skill library faster by working than by writing prompts.

03 — MENTAL MODEL

The five pillars

These five concepts cover everything MudiAgent does. Internalize them and the rest is naming.

01 · MEMORY
Stable facts that survive sessions
MEMORY.md and USER.md load at session start. Auto-extracts facts during work. Update inline ("flush that") at any time. Never holds raw secrets.
02 · SKILLS
Procedural memory
Reusable playbooks stored as SKILL.md files with YAML front-matter. The agent reads your conversations, turns repeated patterns into skills, evolves them with feedback.
03 · SOUL
Personality on top of the brain
SOUL.md shapes voice. Multiple agents, one soul each, all sharing the same memory underneath. Tunes itself as you correct it.
04 · CRONS
Where this leaves your laptop in the dust
Every cron job runs in a fresh isolated session. Flags: CONTEXT, MEMORY, NO_LOOP. Time-bounded crons work to-do loop style. No recursive cron creation.
05 · SELF-IMPROVING LOOP
Why the longer it runs, the better it gets
Do work, get feedback, memory grows, turn steps into skills, search past sessions. Honest caveat: prune and consolidate the skill library, or it bloats.
04 — USE CASES

What we actually run on it

Most of what runs on this stack is marketing infrastructure with the occasional internal tool. The fleet looks like this:

  • Personal assistant. Handles business and private. Lives in Telegram. Flags the four emails worth reading every morning, schedules reminders, summarizes meetings you missed.
  • Workflow prototyping bench. Where new flows (lead magnet, ad creative review, content sprint) get tested against real work for 2 to 3 runs before they get promoted.
  • Specialized agents. SEO, outbound, design review, content writing. Each one with its own soul and scope.
  • Company brain. Monitors Slack, chats, emails, transcripts, voice memos. Makes all of it queryable. "What did we say to that client about pricing last month" returns in three seconds instead of thirty minutes of digging.
  • SEO agent. Runs the full pipeline from keyword seed to published article in one Docker container. 21 steps. No human in the middle until final review.
  • Content distribution. Takes a long-form piece (this guide, for example) and atomizes it across LinkedIn, X, Threads, with platform-specific hooks.
  • Orchestrator. Does not produce work itself. Routes requests to the right specialist based on what you are asking for.
  • Design review. Sees draft ads, hero crops, brand-mark placements, screenshots. Flags what is off-system before it ships.
  • Internal tooler. Builds the small one-off scripts you would otherwise never finish. Plus the cron jobs that catch the silent fail before customers do.

The five layers above the pipeline

  • Layer 1 · Company brain. Vision, brand, audience, products. Every agent reads from this.
  • Layer 2 · Orchestrator. Takes the topic or keyword seed and routes it to the SEO agent.
  • Layer 3 · SEO brain. Ranking playbook, voice rules, content formats, visual style guide, success criteria per format.
  • Layer 4 · Three sub-agents, one per phase: research + ideate, production, distribution.
  • Layer 5 · One Docker container. All three sub-agents share env, memory, and tools. Sub-profiles switch context per phase. One process. One filesystem. One set of credentials.
Why one container instead of three: SEO work is sequential. Research feeds the brief. The brief feeds production. Production feeds distribution. Every step needs memory of what was decided upstream. Splitting into three containers means shuttling state across boundaries, which gets expensive and breaks the chain.

Every other specialized agent runs on the same template. Clone the SEO agent template, swap the brain (SEO brain to outbound brain, or design brain, or support brain), and you have a new agent for any function with the same five-layer shape.

06 — MENTAL MODEL

You, the control room, the agents, the task bus

Before the four levels, the four-part mental model. Holding this in your head makes everything else click.

  • You are the operator. You have direct access to every part of the system.
  • The agent control room is the side control plane. It is not an agent you chat through. It is a folder at /root/vps-agents that documents and governs the whole fleet. You open it, edit it, inspect it, or ask any coding agent on your laptop to use it when you are managing the system.
  • The MudiAgents are the workers. Some are specialists (SEO, dev, CMO, ops). One of them can optionally be an orchestrator.
  • The agent task bus is an optional handoff desk that sits between the orchestrator and the specialists. You only need it once you have an orchestrator in play.

07 — STORAGE SPLIT

Why /root and /srv are different things

The storage split matters more than people think.

filesystem 
/root/vps-agents          → control room: docs, rules, runbooks, architecture
                            no raw secrets, ever

/srv/<agent-name>/data    → live runtime: secrets, memory, skills, sessions, crons
                            this is where each MudiAgent lives

The control room contains the answers to questions like which agents exist, what they do, what ports they use, what credentials they reference, what each agent can and cannot do, and how to restart, debug, or rebuild any of them. The live runtime contains the actual workings.

Core principle

The control room is the brain that defines the system. The live runtime is the body that runs it. You can rebuild the body from the brain. You cannot rebuild the brain from the body.

08 — INTERACTION

Three ways you interact

You have three routes into the system. Each one is for a different job.

interaction paths 
control path:
you ——————▶ agent control room
            (add agents, rotate keys, update docs, debug setup)

direct path:
you ——————▶ mudiagent-seo
            (talk to a specialist directly, fastest)

orchestrated path:
you —▶ mudiagent-orchestrator —▶ task bus —▶ specialists —▶ you
        (one front door, routes and synthesizes multi-agent work)
  • Control path is the meta layer. Used when adding agents, reviewing docs, checking ports, rotating keys, debugging setup.
  • Direct path is the fastest. Used when you already know which agent does the work.
  • Orchestrated path is the synthesizer. Used when you want one front door that routes and combines work across multiple specialists.
09 — LEVEL 1

One agent

You have one MudiAgent. That is it. The control room can still exist (recommended) but it only documents that one agent.

level 1 
you —▶ one MudiAgent

control room —▶ documents that one agent

Best for

Initial setup. Your personal MudiAgent. Root install documentation. Simple Docker migration.

One agent, lived in, with a personality you have tuned and a memory that has started to build. Fill in SOUL.md with the voice you want. MEMORY.md with the stable facts about your business. USER.md with the stable facts about you. Connect it to Telegram or Discord so it lives where you do. Start using it on real tasks. Let it touch your tools. Let it write its own skills along the way.

MEMORY.md holds the stable facts (what your business is, who your customers are, what your products do). USER.md holds the stable facts about you (timezone, working hours, recurring projects, preferred output formats). Both get refined every week as you correct the agent in real conversations.

10 — LEVEL 2

Direct specialist agents

You have multiple specialized agents, but you still talk to each one directly. No orchestrator yet.

level 2 
you —▶ mudiagent-life
you —▶ mudiagent-seo
you —▶ mudiagent-dev
you —▶ mudiagent-cmo

the control room documents all of them.

Best for

Clear role separation. Testing which agents are useful. Avoiding premature orchestration. Keeping credentials scoped per agent.

The trap to avoid here is reaching for an orchestrator before you have proven your specialists are useful. Spin up two or three, run them directly, and only add an orchestrator when you find yourself wanting one front door.

When to spin up a new agent vs. stay with what you have

Bad pattern

One mega-agent with every credential and every memory layer mashed together. You lose isolation. You lose the ability to revoke access cleanly. The agent gets confused about which voice to use.

11 — LEVEL 3

Orchestrator + specialists

You add mudiagent-orchestrator as a front door. You can still talk directly to specialists, but the orchestrator can route work and synthesize results.

The orchestrator reads the control room to know which agents exist, what each one does, where task queues live, what requires approval, which actions are forbidden, and where docs and runbooks are. It does not need to ask you any of that. It reads it.

Best for

Cross-functional work. Delegation. Summary and synthesis. One main interface for multi-agent workflows.

The orchestrator is the moment your setup stops being a collection of agents and starts being a team. It is also the moment where the control room earns its keep, because the orchestrator is only as good as the docs it reads.

12 — LEVEL 4

Automated agent team

Same shape as level 3, but with recurring workflows and stronger automation. Weekly SEO reports run on cron. Server health checks fire daily. Backup verification runs without you asking. Cross-agent business workflows kick off on schedule.

Best for

Weekly SEO reports. Content operations. Server health checks. Backup verification. Cross-agent business workflows.

Level 4 is what a marketing department in your terminal looks like. It does not need you to start the day. It shows up to work on its own, files reports, checks itself, and only pings you for the decisions that need taste.

Control hierarchy

The control room is for config, docs, runbooks, and governance. It is the admin panel for the fleet, including the orchestrator. It is not where you go to do work. For work, you talk to the agents directly. Either a specialist (when you know which agent owns the job) or the orchestrator (when you want one front door to route across specialists).

13 — SETUP

Point your coding agent at the repo. Walk away.

Now you understand the architecture. Here is how you build it.

The exact structure described above lives in a public control-room template, plus the skills the agent needs to set it up for you. You can clone it manually, but the point is that you do not have to. If you have a coding agent on your laptop, the agents do most of the work after you hand over a VPS provider API key.

Within ten to fifteen minutes you have

  • The bundled skills linked into ~/.<coding-agent>/skills on the VPS.
  • One MudiAgent registered, runbook filled in, env-map written.
  • An SSH alias on your laptop so ssh mudiagent connects instantly.
14 — METHODOLOGY

Prototype in MudiAgent. Deploy when it survives. Walk away.

Most workflows do not start as production ones. They start messy. A flow that runs SEO research, drafts an article, schedules it in Typefully, and posts it to LinkedIn does not exist in your head fully formed. You discover it by running it.

MudiAgent is the prototyping environment for this. Here is the four-step path to take any new marketing workflow from idea to autonomous deployment.

  1. Prototype in MudiAgent. Open your main agent, describe what you want to happen, and let it try. It will get most of it wrong on the first run. That is fine.
  2. Run it 2 to 3 times against real work, correcting drift each time. The harness watches every correction and starts writing the skill as it learns the shape. By run three the agent is doing most of what you want without coaching.
  3. Fine-tune in a dedicated workspace. Pull the workflow into a separate coding-agent workspace (or a fresh MudiAgent). Tighten the prompts. Lock the routing. Add error handling. Decide what should run on cron and what should be triggered.
  4. Deploy to a VPS on a schedule. Once it survives a week of real runs without you babysitting it, push it to its own Docker container on your VPS, set the cron, walk away.

Lesson learned the hard way

You cannot write a production agent from scratch. You have to grow one. MudiAgent makes the growing part fast.

15 — MODELS

The framework gives you the shape. The model gives you the taste.

MudiAgent gives you the framework. The model underneath is your choice. Through the tool gateway you can route to 300+ models from one subscription, switching per agent or per task.

What we run today

  • Claude Opus 4.7 for the creative work: copywriting, voice, hook generation, content drafting, anything where taste and writing quality matter.
  • Codex (GPT 5.5 class) for the structured work: coding, planning, multi-step workflows, browser automation, scraping, anything where the steps need to be tight and the output predictable.

Run both. Opus writes. Codex builds and plans. MudiAgent makes routing easy. Point each agent at the model that fits the work it does.

If you can only run one

The answer depends on what kind of work your fleet is doing. Heavy on content and copy? Start with Claude Opus. Heavy on infrastructure, automation, and engineering workflows? Start with Codex. You can always add the second model later through the same tool gateway.

16 — HONEST TRADE-OFFS

Where this still breaks

Three real trade-offs. None of these are dealbreakers, but you should know them before you commit.

01 · The bundled defaults are also opinions

MudiAgent ships with strong defaults for how memory works, how skills get written, how the agent uses tools. That is the whole pitch. But it also means if you want primitives with explicit control over every step, MudiAgent will feel heavy. A primitives framework is the better fit for that taste. Pick the tool that matches your philosophy.

02 · Level 3 and 4 have a real learning curve

Docker, VPS, SSH, the control room folder structure, the orchestrator skills, none of this is "install and go." You should not jump to level 3 if you are not already running MudiAgent at level 1 daily.

03 · The model still matters

MudiAgent is a framework that makes a good model great. It does not make a small model into a strategist. Use the strongest models you can afford for the work that matters (your orchestrator, your strategy agent, your brain). Drop to cheaper models for the work that does not (research scraping, draft generation, batch processing).

None of this is magic. It is a framework that pays back because the memory persists, the skills accumulate, and the agents stay scoped. Apply it to the wrong-sized model and you get a confused team. Apply it to the right one and you get a team.
