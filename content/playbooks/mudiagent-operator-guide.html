The framework

An autonomous agent that gets more capable the longer it runs.

MudiAgent is Muditek's enterprise build of a persistent operator framework. It owns memory that survives between sessions. It writes its own skills as it works. It ships with a curated bundle of operator skills covering GitHub workflows, Obsidian, Google Workspace, Linear, Notion, browser control, web scraping, vision, voice, and scheduling.

It lives wherever it is deployed: a laptop, a Docker container, a VPS, or a serverless runtime. Operators reach it through whatever surface they already use — Telegram, Discord, Slack, email, voice mode, or the terminal itself.

The closed learning loop is what separates this from a smart chatbot. The agent watches itself work, writes new skills as it learns the shape of the operator's work, refines its memory periodically, and recalls past context across sessions.

Two philosophies

Rails vs Linux

MudiAgent is rails. Opinionated defaults. Batteries included. Productive on day one with minimal setup. The agent does more thinking for the operator.

Primitive-first frameworks are Linux. Maximum control, explicit wiring, the agent does exactly what it was told and nothing more. Both are valid — pick the taste that matches the team.

Where MudiAgent earns its keep

  • Persistent memory across every session, queryable across months of work
  • A bundled skill library that grows with use, never shrinks
  • One framework, multiple deployment targets — local, Docker, SSH, serverless
  • Tool-gateway routing to whichever model fits the task (creative work to one model, structured work to another)
  • Twenty-plus messaging surfaces so the operator never has to switch tools to reach it

Architecture

Every agent has three things. Brain. Soul. Skillset.

01 · Brain

Memory that persists

Memory lives at ~/.mudiagent/memories/. Two files, MEMORY.md and USER.md, inject at session start. Voice rubric, brand notes, customer language, last week's corrections — all of it loads before the first prompt. Sessions stored in SQLite, recall is full-text searchable across the entire history.

02 · Soul

Personality on demand

soul.md is where the vibe lives. Concise, sarcastic, blunt, formal, fast or thoughtful. Six agents can share one brain but wear six different souls. One closer-energy outbound rep. One long-form researcher. One terse assistant. The skin is configurable; the underlying state is the same.

03 · Skillset

Bundled, then grown

A curated skill library ships in the box: PR workflows, Obsidian, Google Workspace, Linear, Notion, browser control, web scraping, vision, voice, scheduling. As the agent works, it writes new skills along the way. The library compounds on top of the bundle.

What the agent can talk to

Tool gateway

One subscription routes to many model providers, plus web scraping and browser automation built in. Switch model per agent or per task.

MCP integration

Any external service that speaks the Model Context Protocol becomes a tool the agent can use immediately. No bespoke wrapper needed.

Messaging surfaces

Telegram, Discord, Slack, email, voice, and the CLI itself. The agent meets the operator wherever they already work.

And where the agent can live

  • Laptop — local, fastest to start, lives and dies with the machine
  • Docker container — isolated, portable, the recommended default for fleet work
  • SSH session on a VPS — keeps running with the laptop closed
  • Serverless runtimes (Daytona, Singularity, Modal) — no infrastructure to manage

Operator note

One orchestrator agent reads the company brain, picks the right department, and routes the work. It does not produce output itself — it just makes sure the right specialist sees the right task with the right context.

Deep dive · SEO factory

Five layers. One container. Twenty-one steps.

The SEO agent is the cleanest mapping of the general architecture. Three sub-agents, one shared container, one set of credentials, sequential handoff through memory.

The twenty-one steps

02  serp snapshot
03  competitor extraction
04  intent + format analysis
05  content + visual gap
06  internal validation
07  external validation

[production]
08  angle + positioning brief
09  visual strategy brief

11  draft

[distribution]
16  publish prep
17  schema

19  syndication

Layers above the pipeline

  • Company brain at the top: vision, brand, audience, products. Every agent inherits from this.
  • Orchestrator agent: takes the topic or keyword seed and routes it to the SEO agent.
  • SEO brain: ranking playbook, voice rules, content formats, visual style guide, success criteria per format.
  • Three sub-agents inside the SEO container, one per phase: research + ideate, production, distribution.
  • One Docker container holds all three sub-agents. They share env, memory, and tools. Sub-profiles switch context per phase.
Why one container

SEO work is sequential. Research feeds the brief, the brief feeds production, production feeds distribution. Every step needs memory of what was decided upstream. Splitting into three containers means shuttling state across boundaries, which gets expensive and breaks the chain.

Every other specialized agent runs on the same template. Clone the SEO agent, swap the brain (SEO brain → outbound brain, or → design brain, or → support brain), and a new role exists with the same five-layer shape.

Mental model

From one agent to a full fleet.

Before the levels, the four parts. Internalize these; the rest of this guide becomes navigation rather than learning.

01 · Operator

You

Direct access to every part of the system. The decision-maker, the editor, the only voice that matters when taste is on the line.

02 · Side control plane

Agent control room

A folder at /root/vps-agents that documents and governs the fleet. Opened, edited, inspected, or read by another agent when managing the system.

03 · Workers

MudiAgent specialists

The workers — SEO, dev, CMO, ops, life. One of them can optionally be an orchestrator that routes across the rest.

04 · Optional desk

Agent task bus

An optional handoff desk between the orchestrator and the specialists. Only needed once an orchestrator is in play. Skip until then.

Storage split

Control room and runtime. Brain and body.

The clearest design decision in the whole stack. Get this right and the rest of the system gets easier. Get it wrong and every level after one becomes painful. 

/root/vps-agents → control room: docs, rules, runbooks, architecture
                   no raw secrets, ever

/srv/<agent-name>/data → live runtime: secrets, memory, skills, sessions, crons
                         this is where each MudiAgent lives

The control room contains the answers to questions like which agents exist, what they do, what ports they use, what credentials they reference, what each agent can and cannot do, and how to restart, debug, or rebuild any of them. The live runtime contains the actual workings.

The principle

The control room is the brain that defines the system. The live runtime is the body that runs it. You can rebuild the body from the brain. You cannot rebuild the brain from the body.

Interaction

Three ways an operator talks to the fleet.

control path: you ──────► agent control room (add agents, rotate keys, update docs, debug setup) direct path: you ──────► mudiagent-seo (talk to a specialist directly, fastest) orchestrated path: you ──► mudiagent-orchestrator ──► task bus ──► specialists ──► you (one front door, routes and synthesizes multi-agent work)
Control path

Meta layer

Used when adding agents, reviewing docs, checking ports, rotating keys, or debugging setup. Not where work happens.

Direct path

Fastest

Used when the operator already knows which agent owns the job. No routing hop, no orchestrator latency.

Orchestrated path

Synthesizer

Used when one front door should route and combine work across multiple specialists. Worth the latency for cross-functional tasks.

Levels 1 & 2

Start with one. Earn the right to scale.

Level 1 · One agent

One MudiAgent. That is it. The control room can still exist (recommended) but it only documents that one agent.

you → one mudiagent control room → documents that one agent

Best for: initial setup, the personal MudiAgent, root install documentation, simple Docker migration.

One agent, lived-in, with a personality the operator has tuned and a memory that has started to build. Fill in SOUL.md with the voice. Fill in MEMORY.md with the stable facts about the business. Fill in USER.md with the stable facts about the operator. Connect it to Telegram or Discord so it lives where the work already lives. Start using it on real tasks. Let it touch tools. Let it write its own skills along the way.

Level 2 · Direct specialists

Multiple specialized agents, still talked to directly. No orchestrator yet.

you → mudiagent-life you → mudiagent-seo you → mudiagent-dev you → mudiagent-cmo the control room documents all of them.

Best for: clear role separation, testing which agents are useful, avoiding premature orchestration, keeping credentials scoped per agent.

Trap to avoid

Do not reach for an orchestrator before specialists have proven they are useful. Spin up two or three, run them directly, only add an orchestrator when the operator finds themselves wanting one front door. Premature orchestration is the most common way to ship a fleet that nobody uses.

When to spin up a new agent

  • It needs its own credentials → new agent
  • It needs its own long-term memory → new agent
  • It owns ongoing repeated work that is a separate role → new agent
  • Otherwise, stay with what you have

Levels 3 & 4

From a team to a department that ships on its own.

Level 3 · Orchestrator + specialists

Add mudiagent-orchestrator as a front door. The operator can still talk directly to specialists, but the orchestrator can now route work and synthesize results.

The orchestrator reads the control room to know which agents exist, what each one does, where task queues live, what requires approval, which actions are forbidden, and where docs and runbooks are. It does not need to be told any of that.

Best for: cross-functional work, delegation, summary and synthesis, one main interface for multi-agent workflows.

The earning moment

The orchestrator is when the setup stops being a collection of agents and starts being a team. It is also when the control room earns its keep, because the orchestrator is only as good as the docs it reads.

Level 4 · Automated team

Same shape as level 3, with recurring workflows and stronger automation. Weekly SEO reports run on cron. Server health checks fire daily. Backup verification runs without anyone asking. Cross-agent business workflows kick off on schedule.

Level 4 is what a marketing department in a terminal looks like. It does not need an operator to start the day. It shows up to work on its own, files reports, checks itself, and only pings the operator for the decisions that need taste.

Setup

Point an agent at the control-room template. Walk away.

The template holds the exact structure described above, plus the skills the agent needs to set up the fleet. Manual clone works; the point is that nothing has to be manual. With another agent on the laptop and a Hetzner API key, the plane lands itself.

The automated flow

you ──► generate a Hetzner API key (5 min: sign up, generate a token, drop it in .env) │ ▼ agent ──► create-vps skill spins up a Hetzner box, generates an SSH key, writes the alias to ~/.ssh/config so `ssh mudiagent` works │ ▼ agent ──► setup-control-room skill installs Node, Docker, the agent runtime, then clones the template to the VPS at /root/agent-control-room │ ▼ you ──► finish interactive auth on the VPS │ ▼ agent ──► agent-control-room skill registers the first mudiagent in the docs, fills in the runbook, sets up the env-map │ ▼ you are at level 1 with a documented agent

Within ten to fifteen minutes

  • A fresh Hetzner VPS with the right tooling installed
  • The control room cloned at /root/agent-control-room on the VPS
  • Bundled skills linked into the local skill directory on the VPS
  • One MudiAgent registered, runbook filled in, env-map written
  • An SSH alias on the laptop so ssh mudiagent connects instantly

Methodology

Prototype → production. Grow agents, don't write them.

Most workflows do not start as production ones. They start messy. A flow that runs SEO research, drafts an article, schedules it, and posts to LinkedIn does not exist fully formed in anyone's head. It gets discovered by running it.

MudiAgent is the prototyping environment for this. Four steps from messy idea to autonomous deployment.

Step 01

Prototype in MudiAgent

Open the main MudiAgent. Describe what should happen. Let it try. It will get most of it wrong on the first run. That is fine.

Step 02

Run it 2–3 times

Against real work. Correct drift each time. The harness watches every correction and starts writing the skill as it learns the shape. By run three the agent is doing most of the work without coaching.

Step 03

Fine-tune in a workspace

Pull the workflow into a dedicated workspace. Tighten prompts, lock routing, add error handling, decide what runs on cron versus what runs on trigger.

Step 04

Deploy on schedule

Once the workflow survives a week of real runs without babysitting, push it to its own Docker container on the VPS. Set the cron. Walk away.

The hard-won lesson

Nobody writes a production agent from scratch. The ones that ship are grown. MudiAgent makes the growing fast — but the growing is still the work.

prototype in mudiagent │ ▼ fine-tune in a dedicated workspace │ ▼ deploy autonomous on a vps

Models & trade-offs

Pick the model for the work. Then accept the trade-offs.

Models worth routing

MudiAgent gives the framework. The model underneath is the operator's choice. Through the tool gateway, agents route to many providers from one subscription — switching per agent or per task.

A working default

  • Frontier creative model for copywriting, voice, hook generation, content drafting — anywhere taste matters
  • Frontier engineering model for code, planning, multi-step workflows, browser automation, scraping — anywhere predictability matters

Run both. One writes. The other builds and plans. Point each agent at the model that fits the work it does.

If only one

Pick by workload

Heavy on content and copy? Start with the strongest creative model. Heavy on infrastructure, automation, and engineering workflows? Start with the strongest engineering model.

The other can be added later through the same gateway without touching agent config.

Three honest trade-offs

01 · Opinions

Defaults are opinions

MudiAgent ships with strong defaults for memory, skills, and tool usage. That is the pitch. It also means operators who want raw primitives with explicit control over every step will find it heavy.

02 · Curve

Levels 3 & 4 take learning

Docker, VPS, SSH, the control-room folder structure, orchestrator skills — none of this is install-and-go. Do not jump to level 3 without already running level 1 daily.

03 · Model gap

Framework can't fix model gap

MudiAgent makes a good model great. It does not make a small model into a strategist. Use the strongest models for the work that matters; drop to cheaper models for batch processing and scraping.

None of this is magic. It is a framework that pays back because memory persists, skills accumulate, and agents stay scoped. Apply it to the wrong-sized model and the result is a confused team. Apply it to the right one and the result is a team.
