01 · Foundations

An agent that gets sharper the longer it runs.

Short version: an autonomous agent with memory that survives between sessions and a skill library that grows as it works.

Longer version: MudiAgent turns a model into a persistent operator. It ships with 123 skills already built in: GitHub workflows, Obsidian, Google Workspace, Linear, Notion, Typefully, Perplexity, deep research, and 100+ more. It lives wherever you put it (laptop, Docker, VPS, serverless) and you talk to it through 20+ surfaces.

Why it's different

It routes tasks to the right model by complexity and cost, learns your voice over time, and handles context switching without losing the thread.

MudiAgent vs. a coding assistant vs. agent primitives

If you have used Claude Code or another terminal coding agent, MudiAgent is the same shape with a different philosophy.

Coding assistant
A pair programmer

Lives in your terminal next to your code. Best for sitting down and drafting code with you all day. The day-driver for desk work.

MudiAgent
Rails — productive day one

Opinionated defaults, batteries included. Lighter, faster, focused on the self-improvement loop. Built for work that runs without you watching.

Agent primitives
Linux — explicit control

Primitives and guarantees. The agent does exactly what you told it and nothing more. The right pick if you want control over every step.

Both directions are valid. We run MudiAgent because the bundled defaults compound. Every project starts with the agent already knowing how to do 100+ things before a line of configuration.

02 · The engine

How a single request flows through the system.

Six steps, every part of the engine visible. This is what happens between you asking and the work landing.

1
You ask

You give it a task

CLI, Telegram, Discord, Slack, email, SMS, 20+ surfaces. Plain language, no slash command needed. It runs on a $5 VPS, not your laptop.

2
It scans

It reads the skill index

A one-line description per skill sits in the system prompt at all times. 200 skills is a tiny token cost. 123 bundled, plus your own, plus hub-installed and tapped repos.

3
It loads

It opens the matching SKILL.md

When-to-use phrasing match, the procedure as ordered steps, pitfalls with fixes, and a verification check. Reference files load only if that turn needs them.

4
It stacks context

It pulls what it knows about you

MEMORY.md for agreed facts, USER.md for who you are, full-text search over past conversations, and a dialectic user model. Memory and skills stack, they don't compete.

5
It runs the work

It executes against your environment

Shell, files, git, MCP, web search. Six backends: local, Docker, SSH, Daytona, Singularity, Modal. Serverless options hibernate when idle. Output streams back in real time.

6
It saves what it learned

It nudges itself to persist

"Want me to save this as a skill?" It writes a new SKILL.md with the pitfalls it hit, patches in place when it finds a new edge, and syncs across machines via git.

04 · Skills

How skills work, and why they cost almost nothing.

A skill is a Markdown procedure loaded on demand: a SKILL.md with YAML front-matter (name, description, when-to-use, procedure, pitfalls) plus optional reference, template, and script subfolders.

Where they come from

Bundled · 123+

Ship with every install. GitHub PRs, planning, research, ASCII art, and more.

Agent-authored

MudiAgent offers to save the complex workflows it just solved. Say yes and it persists them.

Hub + taps

Community-published, single URL, GitHub repo. Shared across a team via git.

Progressive disclosure: three levels of loading

Level 0 · Index

Name + description

In the system prompt always. 200 skills is a tiny cost. This is how the agent picks one.

Level 1 · Open

Full SKILL.md

Loads only when the agent decides to use it: when-to-use, procedure, pitfalls, verification.

Level 2 · References

Files on demand

Long tables and vendor docs load only in the turn that needs them. Nothing wasted.

The learning loop

1

A new task arrives

Complex, multi-step, not covered by an existing skill.

2

The agent solves it

Discovering the pitfalls and the order of operations as it goes.

3

It offers to save

"Want this as a skill?" You say yes.

4

It writes the SKILL.md

Next time a similar task appears, the skill is already loaded. It even patches the file as it learns more.

Day one rule

Do not write your own skills on day one. Run real work, let the agent watch, and let the harness write the skills. You build a custom library faster by working than by prompting.

Watch out for: skill explosion and adjacent duplicates. Prune and consolidate the library over time, or it bloats.

05 · Mental model

The five pillars.

Internalize these five and the rest of the system is just naming.

i

Memory

MEMORY.md and USER.md load at session start. The agent auto-extracts durable facts during work. Update inline at any time. Never holds raw secrets.

ii

Skills

Procedural memory stored as SKILL.md files. The agent turns repeated patterns into skills and evolves them with your feedback.

iii

Soul

SOUL.md shapes voice. Multiple agents, one soul each, all sharing the same memory underneath.

iv

Crons

Where this leaves your laptop in the dust. Every job runs in a fresh isolated session. Flags: CONTEXT, MEMORY, NO_LOOP. No recursive cron creation.

v

The self-improving loop

Do work, get feedback, memory grows, steps become skills, past sessions stay searchable. The more you use it, the better it gets.

The pillars are not features. They are the reason it compounds.

The five layers above the pipeline

Layer 1 · Company brain
The context every agent inherits

Vision, brand, audience, products. Stays stable while the worker iterates.

Layer 2 · Orchestrator
Routing

Takes the topic or keyword seed and hands it to the SEO agent.

Layer 3 · SEO brain
Domain context

Ranking playbook, voice rules, content formats, visual style guide, success criteria per format.

Layer 4 · Three sub-agents
One per phase
research + ideateproductiondistribution
Layer 5 · One Docker container
Shared env, memory, tools

Sub-profiles switch context per phase. One process, one filesystem, one set of credentials.

Why one container, not three: SEO work is sequential. Research feeds the brief, the brief feeds production, production feeds distribution. Every step needs memory of what was decided upstream. Splitting it means shuttling state across boundaries, which gets expensive and breaks the chain.

The layers are not decoration. They are the reason the agent does not lose context as the work specializes. The company brain stays stable while the worker iterates, which makes the worker disposable. Clone the template, swap the brain (SEO → outbound → design → support), and you have a new agent with the same five-layer shape.

09 · The fleet

You, the control room, the agents, the task bus.

Before the levels, the four-part mental model. Hold this and everything else clicks.

  • You are the operator, with direct access to every part of the system.
  • The control room is the side control plane: a folder at /root/vps-agents that documents and governs the fleet. Not an agent you chat through.
  • The MudiAgents are the workers. Specialists (SEO, dev, CMO, ops), one optionally an orchestrator.
  • The task bus is an optional handoff desk between the orchestrator and the specialists. You only need it once an orchestrator is in play.

Three ways you interact

Control path
you → control room

The meta layer. Add agents, rotate keys, update docs, debug setup.

Direct path
you → mudiagent-seo

The fastest. Used when you already know which agent owns the job.

Orchestrated path
you → orchestrator → bus → specialists

The synthesizer. One front door routes and combines work across specialists.

The control room governs every agent in this picture. It is the single source of truth and the place you manage the fleet, not the place you run work through.

10 · Storage split

Why /root and /srv are different things.

This split matters more than people think.

filesystem
/root/vps-agents          → control room: docs, rules, runbooks, architecture
                            no raw secrets, ever

/srv/<agent>/data         → live runtime: secrets, memory, skills, sessions, crons
                            this is where each MudiAgent actually lives
Core principle

The control room is the brain that defines the system. The runtime is the body that runs it. You can rebuild the body from the brain. You cannot rebuild the brain from the body.

Inside the control room

/root/vps-agents
/root/vps-agents/
├─ README.md
├─ CLAUDE.md
├─ agents/
│  └─ <agent>/
│     ├─ inventory.md      # ip, hostname, where it runs, who owns it
│     ├─ docker.md         # compose draft + first-boot checklist
│     ├─ env-map.md        # which keys live where, no values
│     ├─ runbook.md        # "it broke, what now" scenarios
│     └─ backup.md         # github backup + hardening checklist
└─ shared/
   ├─ security.md
   ├─ commands.md
   ├─ api-keys-sop.md
   └─ orchestrator-and-fleet-skills.md

11 · Operator levels

From one agent to a full fleet.

01
One agent

Your personal MudiAgent

One agent, lived-in, with a tuned soul and a memory that has started to build. Fill in SOUL.md, MEMORY.md, and USER.md. Connect Telegram or Discord so it lives where you do. Run real tasks. Let it write its own skills. The control room can exist already, documenting that one agent.

02
Direct specialists

Several agents, talked to directly

Clear role separation, credentials scoped per agent, no orchestrator yet. The trap is reaching for an orchestrator too early. Spin up two or three, run them directly, and add routing only when you find yourself wanting one front door.

Spin up a new agent when it needs its own credentials, its own long-term memory, or it owns an ongoing separate role. Otherwise stay with what you have. Avoid one mega-agent with every credential mashed together: you lose isolation and clean revocation.

03
Orchestrator + specialists

The moment it becomes a team

Add an orchestrator as the front door. It reads the control room to learn which agents exist, what each does, what needs approval, and what is forbidden. It does not ask, it reads. This is where the control room earns its keep, because the orchestrator is only as good as the docs it reads.

04
Automated team

A marketing department in your terminal

Same shape as level 3, with recurring workflows. Weekly SEO reports on cron, daily health checks, backup verification without you asking. It shows up to work on its own, files reports, checks itself, and only pings you for the decisions that need taste.

The control hierarchy: the control room is for config, docs, runbooks, and governance. It is the admin panel for the fleet, including the orchestrator. It is not where you do work. For work, you talk to a specialist directly, or to the orchestrator when you want one front door.

13 · Security

API keys, done right.

Treat the agent like a new hire: its own accounts, scoped access, least privilege. Never your personal credit card.

Never paste into chat

Don't drop keys into a container's chat. SSH into the container and set them through config, where they don't land in session logs.

One key per agent

Keys saved to /root/.env/<agent>, named per agent. Revoke one agent without touching the others.

Least privilege

Scoped read access by default. A marketing agent never gets finance credentials.

Never commit secrets

The control room documents which keys live where, never the values. .env stays out of git.

14 · The first cron

The one cron to build first.

Before anything fancy, build the nightly backup. If the VPS dies, your shell and memory die with it. With a backup, you spin up again.

Nightly GitHub sync

"Every night at midnight, push everything except secrets to GitHub."

That single job is the difference between a setback and a disaster. Everything else (weekly reports, health checks, content runs) comes after the thing that lets you rebuild exists.

15 · Methodology

Prototype it. Let it survive a week. Then walk away.

Most workflows start messy. You discover them by running them, not by designing them in your head.

1

Prototype in MudiAgent

Describe what you want and let it try. It gets most of it wrong on run one. That's fine.

2

Run it 2–3 times against real work

Correct the drift each time. The harness writes the skill as it learns the shape. By run three it's mostly right without coaching.

3

Fine-tune in a dedicated workspace

Tighten prompts, lock routing, add error handling, decide what runs on cron vs. on trigger.

4

Deploy to a VPS on a schedule

Once it survives a week without babysitting, push it to its own container, set the cron, walk away.

The lesson

You cannot write a production agent from scratch. You have to grow one. MudiAgent makes the growing fast.

17 · Honest trade-offs

Where this still breaks.

i

The defaults are opinions

Strong defaults for memory, skills, and tool use are the whole pitch. But if you want explicit control over every step, this will feel heavy. A primitives framework fits that taste better.

ii

Levels 3 and 4 have a curve

Docker, VPS, SSH, the control-room structure, orchestrator skills. None of it is "install and go." Don't jump to level 3 if you aren't running level 1 daily.

iii

The model still matters

The framework makes a good model great. It won't make a small model a strategist. Use the strongest model you can afford for the work that matters; drop to cheaper ones for scraping and batch jobs.

None of this is magic. It pays back because memory persists, skills accumulate, and agents stay scoped. Apply it to the wrong-sized model and you get a confused team. Apply it to the right one and you get a team.
