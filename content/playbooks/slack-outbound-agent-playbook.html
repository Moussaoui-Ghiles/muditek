Run Your Entire Outbound From One Slack Channel

Replace ten subscriptions with one open-source agent that has persistent memory, picks the verticals, scrapes the data, validates the emails, and launches the campaigns while you sleep.

Open your outbound stack right now and count the subscriptions. You pay for ten platforms to do one job. The teams moving fastest collapsed all of it into a single Slack channel.

A lead-list tool. An enrichment tool. A verification tool. A CRM. A sequencer. A deliverability monitor. A meeting scheduler. An analytics dashboard. A spreadsheet stitching half of them together. Probably ten platforms, to figure out who to talk to, find a way to reach them, and start a conversation.

The friction is not the tools. The friction is that a human is the integration layer between platforms that do not talk to each other natively. That human spends roughly eight hours a week stitching outputs together by hand. Whatever was run last week gets run again next week, with the same manual stitching in between.

That whole stack collapses into one place: one Slack channel. You DM the agent. It picks the verticals, scrapes the data, validates the emails, drafts the sequences, launches the campaigns, and pings you when it is done. It runs overnight while you sleep.

The reference live build behind this playbook produced 45 verified leads from $0 of paid enrichment, using only public data, in four Slack messages, in roughly one hour.

This document is the install path. Every command, file path, configuration choice, and self-correction rule below is reusable as-is.

1. The core shift

The traditional outbound workflow:

Open Clay. Build a table. Run enrichment columns. Export CSV. Re-upload into Smartlead or Instantly. Switch to a doc to write copy. Switch tabs to QA. Pull up a third tab to check deliverability. Something breaks. Fix it. Re-run.

The tools are not the problem. The human being the integration layer is the problem.

The new workflow:

Open Slack. DM the agent. Wait. The agent returns when the chain is done.

Same outcome. Different operator. The operator now has persistent memory, a sandboxed runtime, and a toolbox. The human becomes the trainer and the reviewer, not the integrator.

2. The architecture: one harness, one brain, many roles

Every use case in this playbook runs on the same four pieces.

  1. One harness. A sandboxed agent runtime with persistent memory on disk. The reference open-source harness is OpenClaw, MIT-licensed, run by an independent foundation that OpenAI helps fund. Any harness with persistent memory plus tool access works in principle. OpenClaw is the most mature option in production right now.
  2. One memory layer. Institutional knowledge of the business lives with the agent, on disk, indexed, walked at session start. Seeded once. Survives context resets.
  3. One interface. Slack is the default. WhatsApp, Telegram, Discord, Google Chat, Signal, iMessage, and Microsoft Teams are first-class alternatives in the harness. Pick where your team already lives.
  4. A skills toolbox per role. Each agent is the same brain with a different system prompt and a different set of tools attached. Outbound gets scrapers and an email-platform CLI. Marketing gets the website repo. Sales intel gets the call-transcript database. Content gets the CMS and the meeting-notes feed.

That is the entire stack. Everything below is execution detail.

One important framing

Read this playbook as proof that the architecture works with substitutions, not as a buyer's guide. Swap the model for whichever is winning this month. Swap the email platform. Swap the chat interface. The pattern stays the same. The substitutions do not break the workflow.

Minimum specs. 2 CPU cores, 4 GB RAM, 10 GB SSD, Node 22 or higher. Bump RAM to 8 GB if browser automation will be enabled, because the underlying Playwright runtime roughly doubles the memory requirement.

Sizing rules. One operator running outbound for a single business runs everything on a Mac mini comfortably. A Mac mini at $799 with 512 GB is currently the entry tier after Apple discontinued the $599 SKU in May 2026, due to AI-driven memory chip shortages. The agent fleet still runs comfortably at that tier.

Stand the harness up first. Train it second. Do not try to do both at the same time.

4. Phase 2. Install and wire to Slack

Three terminal commands followed by an onboarding wizard.

install
npm install -g openclaw@latest
openclaw onboard --install-daemon
openclaw gateway status

The wizard walks through the gateway, the workspace, the channels, and the first skills. For Slack specifically, choose Socket Mode. It gives real-time messaging with no public webhook to expose, which keeps the surface area small and avoids most of the auth pain.

Once the daemon is up, confirm the loop is closed with a single test:

test
openclaw agent --message "Say hi" --thinking high

A reply in Slack means the install is operational.

5. Phase 3. Seed the memory vault

This is the part that determines whether the agent is useful or generic.

The biggest reason agent projects fail is the context window. A session starts. Business context gets loaded for an hour. The limit hits. State evaporates. Tomorrow the ICP gets re-explained again.

The harness flips that. Memory lives on disk, indexed, walked at session start. A 5-file layered system at ~/.openclaw/workspace/memory/.

FilePurpose
MEMORY.mdThe index. Loaded into every session. Points the agent at the other files.
memory/projects.mdThe business. ICP, offers, segments, qualifying signals.
memory/infra.mdTool stack and canonical use per tool. Credentials referenced, never inlined.
memory/lessons.mdCorrections the agent has learned over time. The compounding loop lives here.
memory/YYYY-MM-DD.mdDaily activity log the agent reads back at session start.

Two non-negotiable additions on top of the defaults

  • Memory Protocol. A standing instruction telling the agent to check the memory/ files before answering. Without this, the agent answers from whatever happens to be in the current context window and never consults its notes.
  • Retrieval Protocol. A standing instruction telling the agent how to query the index. Specifies which file to check for which kind of question. Without this, the agent reads everything every time and burns tokens for no gain.

Most setup pain that gets posted in the harness community traces back to missing one of these two.

Highest-leverage extra configuration

A pre-compaction memory flush. The agent senses its context window filling and writes a summary to memory/ before losing state. Without this, the agent forgets the morning by the afternoon.

Block one afternoon to write the actual content. Treat it like onboarding a senior hire. The minimum vault content:

  • ICP definition. Segments. Sub-personas. Qualifying signals for each.
  • Top three customer pains, in customer language. Pull verbatim from real call transcripts. Not marketing copy. Their words.
  • Offers and pricing. Including which offer maps to which segment.
  • Competitor positioning. Who is lost to, who is beaten, and the one-line reason for each match-up.
  • Winning sales narratives. The three to five storylines that have closed real deals. Plus the objections that come with each.
  • Post and email formats that already work. Specific subject-line patterns, hook formats, CTA structures, post structures.
  • Historical wins. Past clients, deal sizes, segments. The agent uses this to rank future verticals.

A few days of investment to seed it. The payoff compounds every week after that.

6. Phase 4. Add skills, one at a time

The harness ships skills in three layers.

LayerToolsDefaultWhat to do
Core8 toolsOnFile access, command execution, web access. Do not touch.
Advanced18 toolsOffBrowser control, sub-agents, scheduled automation. Enable as needed. Never preemptively.
Knowledge53+ skillsOffSlack, Google, Obsidian, HubSpot, and so on. Install only what is actually used.

Each skill lives at ~/.openclaw/workspace/skills/<skill>/SKILL.md. The description field in that markdown directly controls trigger probability. Be specific and list the actual phrases the team uses.

A specific description triggers reliably:

"Launch an Instantly campaign from a list of verified leads."

A vague description gets misfired or ignored:

"Help with email."

7. Phase 5. Run one workflow three times before adding the next

This is where most teams self-sabotage. The temptation is to deploy a marketing agent, a sales agent, an outbound agent, and a content agent in week one. That guarantees a month of debugging instead of compounding.

The rule:

  1. Pick the single workflow with the most repetitive manual work.
  2. Have the agent own it end-to-end.
  3. Watch it run once. Correct what it gets wrong, in plain English, in Slack. Confirm the correction lands in memory/lessons.md.
  4. The second pass should need only a small nudge.
  5. By the third run, the workflow should be hands-off.

After the first agent is stable, the two highest-value next builds are usually sales-call coaching (high leverage, low visibility) and a marketing agent with GitHub PR access on the website (lower leverage, high visibility). A content-drafting agent is the slowest-burn option but compounds the most consistently over time.

8. Phase 6. Harden before client data touches it

A standard install runs as a privileged process on the host. Before any client data or production credential is pointed at the agent, run the hardening pass.

The 12-step hardening checklist

  • Firewall rules on the gateway port. Only the bare minimum exposed.
  • Gateway-level authentication. No anonymous access to the agent endpoint.
  • Tailscale or equivalent for private networking between operator and host.
  • Execution-approval controls on shell commands. No destructive commands without explicit approval.
  • Credentials stored in a secrets manager. Not in memory/infra.md. Not in committed env files.
  • Audit logging on every agent action. Tool calls, file edits, network requests, all of it.
  • Explicit allowlist on the skills the agent is permitted to call. No "all skills always available."
  • Per-tenant memory isolation if the same harness serves multiple clients.
  • Network egress rules on the host. Block outbound traffic to anything not allowlisted.
  • Filesystem isolation between memory vaults of different agents in a fleet.
  • Versioned backups of the memory vault. Daily, off-host.
  • A documented rollback path for a misbehaving agent, including credential revocation in under a minute.

For regulated industries (healthcare, financial services, anything HIPAA or SOC 2 adjacent), skip the standard install and start with a hardened distribution.

NemoClaw is NVIDIA's hardened distribution of OpenClaw, released in early preview on March 16, 2026. It wraps the harness with:

  • Full local inference via Nemotron-class open models.
  • Process-level sandboxing through NVIDIA's OpenShell runtime.
  • Policy-based privacy and security guardrails.
  • Network and filesystem isolation by default.

Zero data leaves the device. The performance ceiling is lower than cloud frontier models, but for regulated buyers that tradeoff is the point.

9. Phase 7. The free-data outbound recipe

The reference live build that produced 45 verified leads from $0 in paid enrichment ran this exact sequence. Reuse the steps as-is.

Step 1. Vertical selection

Prompt the agent to find high-ticket verticals where the buyers likely do not have an engineering bench, using only public data. The output is a ranked tier list shaped by the historical wins in the memory vault:

  • Tier 1, highest conviction. Verticals already closed before, with a clear automation gap.
  • Tier 2, strong fit, more competitive. Same gap, bigger incumbents.

If the ranking reads like a generic LLM guess, the memory vault is under-seeded. Add more historical wins and re-run.

Step 2. Mine pain in customer language

Prompt the agent to scrape public review sites for pain signals in the chosen verticals. The right sources are G2 and Capterra. Buyers explain in writing why they bought, why they switched, and what is broken. The output is verbatim user complaints, vertical by vertical, product by product.

Each complaint becomes an outbound trigger. A complaint about a specific incumbent ("expensive, buggy, missing core features") is a hook. The hook goes into the sequence verbatim, in the buyer's own words.

If a public source is blocked on the day of the run (Reddit's API walled off, for example), let the agent pivot on its own. Do not redirect. The agent should explain the pivot in the Slack thread in plain English. The operational lesson is one line:

Let the agent solve. Do not micromanage.

Step 3. Pull volume from public directories

Prompt the agent to retrieve company lists from public directories for the chosen verticals. Reference live-build numbers from a single overnight run:

  • 4,966 financial advisory firms.
  • 15,000+ electrical contractor businesses.

Public sources to chain together: G2, Capterra, Google Maps, public business directories, company contact pages, sitemap-driven page-scraping for "About" and "Contact." Volume is not the limit on free-data prospecting. Quality control is.

Step 4. The DNS MX check rule

This is the rule worth memorizing. It is what makes free-data prospecting actually viable. Write it into the agent's memory once.

The rule

1. Pull the company domain. 2. Run a DNS MX check on the domain. 3. If there is no MX record, the domain cannot receive email, so drop the row. 4. Only after the MX check passes, scrape the contact page or guess the email format.

This rule cuts most of the wasted work out of free-data prospecting before any enrichment credit is spent. It is free. It runs in milliseconds. It is the difference between a usable list and a junk list.

Step 5. Overnight run, async return

Prompt the agent to grind overnight and notify in Slack when the batch finishes. Close the laptop. No polling. The reference live build cleared 45 verified leads on the first pipeline. Real names, real phone numbers, placeholders filtered. The agent posted a Slack message when the batch was ready.

Step 6. Launch the campaign from inside Slack

Prompt the agent to draft the campaign and push it live. Reference parameters from the live build:

  • Campaign naming. Today's date plus the segment label. Makes future reruns and audits trivial.
  • Cadence. 3-step sequence at day 1 / day 4 / day 9.
  • Volume. 10 emails per day per inbox times 10 secondary domains equals 100 emails per day from cold start.
  • Variables. Company name, first name, vertical-specific pain hook lifted from the G2 signal in Step 2.
  • Delivery. The agent shells out to the email-platform CLI to create the campaign programmatically.

Three messages on Slack and the operation is live. The operator did not open the email platform's dashboard once.

10. The self-correction loop

Every mistake the agent makes once becomes a rule in memory. The next run does not repeat it. The run after that runs even cleaner. Over weeks, the system audits itself into something close to consistent.

Rule A. API variable casing

The agent created its first campaign with company_name (snake_case) where the platform API expects companyName (camelCase). The operator flagged it once in Slack. The agent wrote the rule into memory/lessons.md in the same response. From that point forward, every campaign uses the correct casing automatically.

Generalize it. When an API returns a 400 because of a variable-shape mismatch, the agent writes the correct shape into memory, scoped to that integration, before retrying.

Rule B. Spam-trap filtering

The first lead pull surfaced some placeholder emails. The agent wrote a memory rule:

"Spam-trap filter now baked into pipeline for all future batches."

The next pull was cleaner. The one after that, cleaner again.

Generalize it. Any data-quality regression caught on review becomes a permanent pre-output filter for the same pipeline stage. The agent does not re-introduce the regression.

Without this loop, every batch starts over. With it, every batch starts ahead of the previous one.

11. CLI-first, MCP as a fallback

The tooling layer is the under-discussed reason the install scales.

The protocol most teams default to is MCP. It is the protocol Anthropic shipped and that most agent frameworks adopted. It is still the right answer when a third party ships only an MCP server and there is no CLI alternative.

When there is a choice, build the tooling layer as CLIs first and reach for MCP second. The argument:

  1. Agents already run in a terminal. Every coding-agent session is making bash calls anyway. CLIs fit that environment natively with nothing to translate.
  2. CLIs are cheap to install. A single npm install. The agent can install them itself when needed.
  3. CLIs do not consume the agent's context window the way persistent MCP server connections do. Fewer tokens burned per task. Lower cost, higher throughput, longer effective sessions.
  4. CLIs have decades of history being chained, scripted, and composed. Logs, pipes, exit codes, redirects, cron. All of it just works.

The reference instantly-cli covers 156 commands across 31 API groups, each command runs as a bash call, and each is paired with a skill file that describes commands, parameters, and common failure modes. The agent installs, reads the skill, executes. Accuracy stays high because the surface area each tool exposes is narrow and explicit.

The same binary doubles as an MCP server. One install. Two surfaces. The CLI is the default path. The MCP server stays available for clients that need it. Reuse the pattern for any tool: ship a CLI that also exposes an MCP surface.

12. The fleet pattern: four roles, one architecture

One outbound agent is the entry point. The same architecture runs the rest of the GTM stack. Four reference roles, all on one harness.

A. Marketing agent

  • Access. The website repository, with commit rights on a feature branch only, never on main.
  • Trigger. A Slack message ("ship a new pricing page," "add a holiday banner," "update the meta description," "write a case-study landing page").
  • Flow. The agent generates the revision on a new branch, opens a pull request, creates a deploy preview, and posts the preview link back to Slack. The operator reviews in the browser and merges in the channel.

Functionally a chief marketing officer who actually has the codebase. Most agencies hire two people to do less than this.

B. Sales intelligence agent

  • Access. The call-recording database. Supabase is the reference. Any database that can ingest transcripts works.
  • Pipe-in tools. Any call-recorder that writes to the database reliably. Fireflies and Granola both work. The recorder choice is replaceable.
  • Trigger. Every call recording lands in the database, tagged by call type.
  • Flow. The agent pulls the transcript, identifies the opportunity, scores it against the offer, posts a Slack summary, and then coaches the rep on what to change next call ("on minute 14 you missed the budget signal; on minute 22 the prospect referenced a competitor and you did not probe"). When the call closes well, the agent drafts the proposal. The operator approves. The agent sends.

C. Content agent

  • Access. The memory vault plus source material from real calls and internal Slack threads.
  • Trigger. Something interesting (a customer outcome, a product update, a sharp insight from a meeting).
  • Flow. The agent drafts the post in the operator's voice, grounded in the source material, and queues it for review. The drafts are concrete because the source material is concrete. Real outcomes, not generic best-practice content.

D. Outbound agent

Everything in Section 9. Same brain. Different system prompt. Different toolbox. One Slack workspace.

13. Two operating rules for working with the fleet

These are non-obvious. Both come out of real operator practice.

Rule 1. Talk to the agent. Do not type.

Voice-to-text gives the agent more context per message than typed text. People ramble when they talk. The rambling is the context. Typing throttles the operator to bullet points and strips the reasoning the agent could have used. Use voice for any non-trivial instruction. Use typing for one-line nudges.

Rule 2. Give every agent a name, a persona, and a backstory.

When the team starts treating the agent as a teammate, the workflow changes. People delegate. People follow up. People hold the agent accountable the way they hold a junior hire accountable. The system prompt for each agent should include the name, the role in one line, a short backstory, and a tone. The output reads like a colleague.

14. Model selection inside the fleet

Different brains for different jobs. The pattern that works in practice:

  • Sonnet-class models for the workhorse roles. Outbound research, lead pulls, campaign drafting, content drafting, marketing PR generation. Throughput beats marginal accuracy.
  • Opus-class models for the harder reasoning. Sales-intel scoring, proposal generation, strategic call coaching, vertical ranking when the memory vault is rich. Accuracy beats throughput.

The routing happens in the personalities and the channels, not in a config file. The Slack channel structure does the routing. Outbound DMs go to the Sonnet-class agent. Sales-intel summaries come from the Opus-class agent.

15. The agency model inversion

The work used to be generating the leads. Building lists. Enriching. Writing sequences. Monitoring inboxes. Reporting meetings booked at the end of the week. That work is heading to zero faster than most teams realize. Inside 12 months, most of it will be a single agent's job.

The work now is installing the system that does all of the above and keeping it smart as the models keep upgrading underneath it.

Clients are dropping the request for "lead generation." What they actually want is an agent that generates leads on its own forever, with someone on call to keep it running.

The SaaS layer that did one specific thing well (list-building, enrichment, sequencing, scoring) is becoming a commodity input the agent calls when it needs it. The value moves up the stack to whoever owns the agent and trains it. The next year of durable outbound offers will look like this:

  • One-time install of the harness, the memory vault, and the fleet.
  • Monthly retainer to keep the system trained as the data, tools, and models evolve.
  • Outcome reporting (meetings, pipeline, revenue) instead of activity reporting (emails sent, lists built).

Sell the operator, not the labor.

18. A short list of things to memorize

  • One channel. One brain. One toolbox per role. That is the entire architecture.
  • The memory vault is the asset. Seeding it is the work. The harness is a weekend.
  • DNS MX check before any enrichment. Free-data prospecting only works because of this.
  • Pain mining lives on G2 and Capterra, not Reddit.
  • Memory Protocol plus Retrieval Protocol are non-negotiable. Without both, the agent ignores its own notes.
  • Pre-compaction memory flush is the single highest-leverage extra configuration.
  • CLI greater than API greater than MCP greater than browser. Default to the cheapest surface that works.
  • SKILL.md description controls trigger probability. Write the actual phrases the team uses.

  • Run one workflow three times before adding the next. Stability before fleet expansion.
  • Harden before client data touches it. The 12-step pass is the gate.
  • Sell the operator, not the labor. This is the new shape of the deal.

20. When this is not the right answer

This playbook assumes:

  • The ICP, the top three customer pains, and the winning narratives can be articulated. If they cannot, no agent fleet rescues the GTM. Fix the clarity first.
  • Cold email is legal in the target geography for the segment being approached. The agent does not fix a compliance problem.
  • One day of senior-operator time is available to seed the memory vault. The compounding does not start until that day is done.
  • The team is willing to review the first three runs of any new workflow. The "hands-off by run three" pattern depends on the first three corrections landing in memory cleanly.

If those four are true, every step above is reusable as-is.

What this playbook deliberately does not invent

To keep every instruction trustworthy, this playbook only includes commands, file paths, and configuration that exist in working agent deployments today. It does not include:

  • Guaranteed lead volumes or reply rates. Outcomes vary by segment, list quality, and offer. Results are not typical.
  • A guarantee that any specific public source stays scrapable. Source policies change. The MX rule and the pivot behavior are the durable parts.
  • A guarantee that prompts behave identically across model versions. Test small, then scale.

If you choose to run these steps, you are responsible for vetting each stage and the legality of cold outreach in your target geography.
